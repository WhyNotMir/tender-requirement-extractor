# Tender Extractor — рабочий журнал

Извлечение тендера (1..n PDF) в трёхуровневое дерево требований `ProcurementMatchDeliverable`
(по заданию BOND). Это **не** финальный README, а рабочий журнал: после каждого слайса
сюда добавляется запись на русском — что сделали, зачем, какие файлы и функции участвуют,
какой путь данных внутри модуля и как он связан с остальными.

Соглашения проекта: код и логи — на английском; комментарии в коде — минимальные, только
там, где это действительно нужно. Этот журнал — на русском.

---

## Слайс 0 — каркас проекта и инфраструктура ключа DeepSeek

**Зачем.** Заложить пустой, но запускаемый скелет и сразу безопасно хранить ключ DeepSeek
вне кода, чтобы случайно не закоммитить его в git.

**Что сделали (файлы).**
- `package.json`, `tsconfig.json` — Node + TypeScript (ESM, строгий режим). Запуск через
  `tsx`, без сборки. Стиль: `await` без `.then()`.
- `.gitignore` — не коммитим `node_modules/`, `env/.env`, `output/`, логи.
- `env/.env.example` — шаблон переменных (**коммитится**, без секретов): `DEEPSEEK_API_KEY`,
  `DEEPSEEK_BASE_URL`, `DEEPSEEK_MODEL` и закомментированный `DATABASE_URL` на будущее.
- `env/.env` — реальный файл с ключом, **в .gitignore**. Сюда ты вставляешь ключ DeepSeek.
- `backend/src/index.ts` — временная точка входа. Функция: грузит `env/.env` через `dotenv`
  по абсолютному пути и печатает, найден ли ключ (замаскированно) и какая модель. Логики
  извлечения пока нет — это проверка, что проводка окружения работает.
- `frontend/` — пустой каркас под будущий UI ревью требований (по заданию не обязателен,
  закладываем структуру заранее).

**Путь данных при запуске.**
`npm start` → `tsx` запускает `backend/src/index.ts` → `dotenv` читает `env/.env` →
в консоль выводится статус ключа и модели. Всё.

**Как запустить.**
```
cd tender-extractor
npm install
# впиши ключ в env/.env  ->  DEEPSEEK_API_KEY=...
npm start
```

**Проверка.** `npm run typecheck` проходит чисто; `npm start` печатает `[boot] ... skeleton OK`
и корректно определяет наличие ключа.

**Связь с дальнейшим.** `index.ts` на следующих слайсах превратится в оркестратор пайплайна
(ingest → extract → … → validate). Конфиг ключа из `env/.env` понадобится начиная со слайса
с реальным DeepSeek. Следующий слайс — логгер (JSON-логи) и парсер аргументов CLI.

## Слайс 1 — логгер и конфиг CLI

**Зачем.** Заложить два инфраструктурных кирпича, на которых будет стоять весь пайплайн:
единый формат логов (бриф прямо просит «we read your logs») и типизированный разбор
аргументов командной строки. Пока без PDF.

**Что сделали (файлы и функции).**
- `backend/src/logger.ts` — структурный логгер. Экспортирует `log.info/warn/error(stage, msg, data?)`.
  Каждый вызов печатает **одну JSON-строку**: `ts`, `level`, `stage`, `msg` и опциональный `data`.
  `error` идёт в `stderr`, остальное — в `stdout`. Единый формат = логи легко читать и парсить.
- `backend/src/config.ts` — конфиг запуска. Функция `parseArgs(argv)` превращает флаги
  `--input <папка> --output <папка> --use-llm` в типизированный объект `RunConfig`
  (`inputDir`, `outputDir`, `useLlm`). Если `--input` не задан — понятная usage-ошибка.
  Здесь же централизованно грузится `env/.env` (перенесли из `index.ts`) и экспортируется `envPath`.
- `backend/src/index.ts` — ужался до оркестратора-заготовки: парсит конфиг и пишет один
  стартовый лог (папки ввода/вывода, режим LLM, найден ли ключ DeepSeek).

**Путь данных при запуске.**
`npm start -- --input <папка>` → `parseArgs` читает флаги → `index.ts` логирует старт
через `logger`. Нет `--input` → бросается usage-ошибка и процесс падает с ненулевым кодом.

**Как запустить.**
```
npm start -- --input "../путь-к-папке-с-PDF" --output output
```

**Проверка.** `npm run typecheck` чист; с `--input` печатается JSON-лог `stage:"run"`;
без `--input` — usage-ошибка.

**Связь с дальнейшим.** `logger` будет вызываться из каждой стадии (свой `stage`),
`RunConfig` пойдёт в оркестратор. Следующий слайс — **ingest**: чтение папки с PDF и
построение манифеста (id файла, SHA-256, число страниц, догадка о роли).

## Слайс 2 — ingest (манифест файлов)

**Зачем.** Первый шаг пайплайна: превратить папку тендера в **манифест** — список файлов
с устойчивыми id и контрольными суммами. Это фундамент провенанса (позже chunk-id ссылаются
на `fileId`) и точка, где в будущем будем ловить «аннекс упомянут, но отсутствует».

**Что сделали (файлы и функции).**
- `backend/src/types/internal.ts` — первые рабочие типы: `FileEntry` (`fileId`, `filename`,
  `pages`, `sha256`, `roleGuess`, `language`) и `Manifest` (`tenderId`, `createdAt`, `files`).
- `backend/src/pipeline/ingest.ts` — функция `ingest(inputDir)`:
  - `readdir` папки, берём только `*.pdf`, сортируем;
  - `fileIdFromName` — стабильный id из имени (нижний регистр, не-буквенно-цифровые → `_`);
  - `sha256` каждого файла (для аудита/дедупа);
  - `guessRole` — эвристика роли по имени (`main_lv` / `annex` / `notice` / `unknown`);
  - `pages` пока `0` — заполнит стадия extract.
  Возвращает `Manifest`. Логирует «found N PDF(s)» и по строке на каждый файл.
- `backend/src/index.ts` — теперь обёрнут в `try/catch` (top-level `await`, без `.then/.catch`):
  парсит конфиг → зовёт `ingest` → логирует итог. Любая ошибка → один `error`-лог и выход с кодом 1.

**Путь данных.** `--input <папка>` → `ingest` читает PDF-и → на каждый считает SHA и роль →
собирает `Manifest` → оркестратор логирует `tenderId` и число файлов.

**Проверка.** `npm run typecheck` чист. На трёх примерах тендеров ingest находит 3 PDF,
присваивает `fileId`, считает SHA (детерминированный), роль — `main_lv` у всех трёх.

**Связь с дальнейшим.** `Manifest.files` пойдёт в стадию **extract**: по каждому файлу
достаём текст с координатами и число страниц (заполним `pages`), плюс оценка качества текста.

## Слайс 3 — extract (текст из PDF + оценка качества)

**Зачем.** Первый слайс, где реально читаем PDF. Достаём текст **с координатами** (нужно
позже для восстановления структуры и чистки колонтитулов) и сразу проверяем качество
текстового слоя, чтобы битые страницы не поехали дальше в LLM как мусор.

**Что сделали (файлы и функции).**
- Зависимость `pdfjs-dist` (чистый JS, портируемый; берём text-слой, не рендерим).
- `types/internal.ts` — новые типы: `Line` (строка с `page/y/x/text`), `PageKind`
  (типы страниц на будущее), `PageInfo` (`textQuality`, `charCount`, `method`).
- `pipeline/extract.ts` — функция `extract(inputDir, file)`:
  - `reconstructLines` — собирает элементы текста в строки: группировка по `y` (допуск 3),
    порядок слева-направо по `x`, склейка с пробелом по разрыву координат;
  - `scoreQuality` — оценка страницы: `clean` / `corrupted` / `image_only`. `corrupted`
    ловится по «сломанной лигатуре» (буква + пробел + `on/ng/ed/...`, как «Installa on»,
    «tes ng») и по артефактам (`Æ`, replacement-символ);
  - на каждую битую страницу — `warn`-лог; в конце — `info` с числом страниц и битых.
  Возвращает `ExtractedFile` (`pages`, `lines`, `pageInfo`).
- `index.ts` — цикл по `manifest.files`: зовёт `extract`, заполняет `file.pages`,
  логирует итог (файлы + суммарные страницы).

**Путь данных.** manifest → по каждому файлу `extract` тянет текст с координатами → строки
+ `PageInfo` с оценкой качества → `pages` записывается в манифест.

Отдельно: `pdfjs` печатает служебные предупреждения о полифилле `DOMMatrix`/`Path2D`
(для чтения текста не важны). Мы их точечно глушим, чтобы поток логов оставался чистым JSON;
наши собственные warn-логи про битые страницы при этом сохраняются.

**Проверка (на 3 примерах).** Fahrrad — 6 страниц, 0 битых; Salzburg — 409 страниц, 0 битых
(за ~1 сек); Christmas — 5 страниц, **2 битые (стр. 2–3)** с warn-логом. Итого 420 страниц.
Никаких `Warning: Cannot polyfill ...` в выводе быть не должно.

**Связь с дальнейшим.** `lines` и `pageInfo` пойдут в стадию **classify**: снятие
колонтитулов (строки, повторяющиеся на многих страницах) и определение типа каждой страницы.

## Слайс 4 — classify (снятие колонтитулов + типы страниц)

**Зачем.** Отделить «мусор» (колонтитулы, адрес планировщика, «Übertrag», «Seite n von N»)
от содержательного текста и пометить каждую страницу её типом — чтобы дальше парсить только
нужное и не тащить в требования шум.

**Что сделали (файлы и функции).**
- `pipeline/classify.ts`, функция `classify(lines, pageInfo)`:
  - `norm` — нормализация строки (цифры → `#`), чтобы повторяющийся колонтитул схлопывался
    в один ключ независимо от номера страницы;
  - `detectChrome` — строки, чей нормализованный ключ встречается на ≥40% страниц (но не
    меньше 3), помечаются как chrome и вырезаются из контента;
  - `classifyPage` — тип страницы по ключевым словам и структуре: `toc` / `preamble` /
    `lv_positions` / `recap` / `prose_spec` / `form` / `unknown`;
  - логирует число снятых строк и разбивку по типам.
- `index.ts` — в цикле после `extract` зовёт `classify`; `pageInfo.kind` проставляется на месте.

**Путь данных.** строки+`pageInfo` из extract → `detectChrome` убирает повторяющийся chrome →
`classifyPage` тегирует страницы → возвращается `contentLines` (чистый текст) для следующей стадии.

**Найденный и исправленный баг.** Recap-регекс `ust\b` (австрийский НДС «USt») ошибочно ловил
английское «m**ust**», из-за чего страницы Christmas уходили в `recap`. Заменили на `\bust\b` —
Christmas стал корректно определяться как `prose_spec`, а у Salzburg почистились ложные recap.

**Проверка (на 3 примерах).**
- Fahrrad: снято 26 строк chrome; типы `{toc:1, preamble:1, lv_positions:3, recap:1}` — идеально.
- Salzburg: снято **6061** строк colontitul/Übertrag; типы в основном `unknown:377` (room-коды
  ещё не парсятся — это ждёт слайс grammar), плюс `preamble:4, recap:15, toc:1, form:12`.
- Christmas: `{form:1, prose_spec:1, unknown:3}` — спец-страница распознана как проза.

**Связь с дальнейшим.** `contentLines` пойдут в стадию **parseLv**: детерминированный разбор
позиций по грамматике номеров (сначала GAEB `NN.NN.NNNN`), из которых родятся чанки-требования.

## Слайс 5 — контракт `ProcurementMatchDeliverable` + Zod

**Зачем.** Зафиксировать **внешний контракт** — точную форму узла дерева из задания BOND —
как TypeScript-тип и как рантайм-схему. Тип ловит ошибки на этапе компиляции, Zod-схема
проверит уже собранное дерево на этапе validate. Это отдельный файл, чтобы контракт с BOND
не смешивался с нашими внутренними рабочими типами (`internal.ts`).

**Что сделали (файлы).**
- Зависимость `zod` (рантайм-валидация).
- `types/procurement.ts`:
  - `LocaleObject<T>` — карта «код языка → строка» (напр. `{ de: "..." }`);
  - `ProcurementMatchDeliverable` — интерфейс **дословно из брифа**: `bulletPoint`,
    `description`, `priority`, `confidence`, `equivalenceAllowed`, downstream-поля
    (`fullfillable`, `status`, `aiReasoning`, `feedback`, `openQuestionId`, cited-массивы),
    `deliverableArray` (дети) и `procurementDocumentChunkIdArray` (источники);
  - `downstreamDefaults` — готовые empty/null значения для полей будущих стадий BOND
    (их бриф велит просто занулить);
  - `DeliverableSchema` — рекурсивная Zod-схема (через `z.lazy`, т.к. узел содержит сам себя),
    зеркалит интерфейс.

**Проверка.** `npm run typecheck` чист. Отдельной проверкой: валидное вложенное дерево
(L1 → L3) проходит `DeliverableSchema.safeParse` (`success:true`), а узел с `priority:"critical"`
отклоняется с сообщением «Expected 'must' | 'should' | 'optional'». Схема ещё **не подключена**
в пайплайн (нечего валидировать) — её задействуют стадии assemble/validate.

**Связь с дальнейшим.** `ProcurementMatchDeliverable`, `downstreamDefaults` и `DeliverableSchema`
будут использованы в стадии **assemble** (сборка дерева) и **validate** (проверка схемы + покрытия).

## Слайс 6 — parseLv (грамматика позиций GAEB)

**Зачем.** Ядро подхода «structure-first»: структура сметы закодирована в номерах позиций,
поэтому мы восстанавливаем требования **детерминированно, по грамматике**, ещё до всякого LLM.
Это даёт высокий recall и почти ноль выдумок — грамматика не может «придумать» позицию.

**Что сделали (файлы и функции).**
- `types/internal.ts` — типы `ChunkKind` и `Chunk` (единица источника: `id`, `ozCode`,
  `title`, `text`, `quantity`, `equivalence`).
- `pipeline/parseLv.ts`, функция `parseLv(fileId, contentLines)`:
  - грамматика GAEB: `RE_TITLE` (`01`), `RE_GROUP` (`01.01`), `RE_POSITION` (`01.01.0010`);
  - идёт по строкам в порядке чтения, копит тело позиции до следующего маркера (`flush`);
  - `RE_QTY` вытаскивает количество (`2 St`, `1 psch`); «oder gleichwertig» → флаг `equivalence`;
  - строки-итоги (`Summe/Zusammenstellung/Übertrag`) и пустые ценовые «......» отбрасываются;
  - пустышки «Entfällt» помечаются `void` и не станут требованием;
  - `id` чанка = `{fileId}:p{page}:{ozCode}` — стабильный и читаемый (провенанс).
  Возвращает чанки + карты `titles`/`groups` (для будущей сборки дерева).
- `index.ts` — в цикле передаёт `contentLines` из classify в `parseLv`, копит `allChunks`,
  логирует общее число позиций.

**Путь данных.** classify → `contentLines` → `parseLv` разбирает номера позиций →
чанки-кандидаты (`kind:"position"`) + карты заголовков/групп.

**Проверка (на 3 примерах).**
- Fahrrad — **6 позиций**, 2 title, 2 group, 1 с «oder gleichwertig». Идеально.
- Salzburg — **1 позиция**: тут номера вида `GU.20.05.01.01` (room-coded), а GAEB-грамматика
  их не ловит. Это **ожидаемо** — грамматика Salzburg будет отдельным слайсом (крупнейший
  прирост recall). Число titles/groups (16/8) — побочные срабатывания на числовых строках, безвредны.
- Christmas — **0 позиций**: чистая проза без номерной грамматики; её путь — извлечение по
  заголовкам/спискам (позже, через LLM).

**Связь с дальнейшим.** Чанки-позиции пойдут в стадию **leaves**: превращение чанка в
кандидат-лист (bulletPoint/description/priority/...) через интерфейс LLM-провайдера
(по умолчанию — офлайн-stub). Карты `titles`/`groups` дадут уровни L1/L2 при сборке дерева.

## Слайс 7 — leaves + интерфейс LLM-провайдера + офлайн-stub

**Зачем.** Превратить чанки-позиции в кандидат-листья (поля будущего дерева) и заодно
зафиксировать **границу с LLM**: весь код общается с моделью только через один интерфейс.
По умолчанию работает детерминированный офлайн-провайдер — пайплайн запускается без ключа
и без сети, а реальный DeepSeek подключится позже той же формой, не трогая стадии.

**Что сделали (файлы и функции).**
- `types/internal.ts` — тип `CandidateLeaf` (лист до вставки в дерево: bulletPoint,
  описание, priority, equivalence, confidence, `chunkIds`, ключи/подписи уровней L1/L2).
- `llm/provider.ts` — интерфейс `LlmProvider` с двумя методами: `enrichLeaf` (обогатить чанк
  в поля листа) и `verifyMerge` (подтвердить слияние при консолидации). Плюс типы
  `EnrichInput`/`EnrichResult`.
- `llm/stub.ts` — `StubProvider` (детерминированный, офлайн): `bulletPoint` — из заголовка,
  `description` — **дословная копия** исходного текста (ничего не выдумывает), `priority` —
  по явным маркерам (`muss/must` → must, `soll/möglichst` → should, `ca./optional` → optional),
  `equivalence` — по флагу «oder gleichwertig», `confidence` — high при заголовке и длинном теле.
  `verifyMerge` всегда `false` (спекулятивные слияния не одобряет).
- `pipeline/leaves.ts`, функция `buildLeaves(...)`: по каждой позиции берёт ключи уровней из
  OZ-кода (`01` → L1, `01.01` → L2), тянет подписи из карт `titles`/`groups`, зовёт
  `provider.enrichLeaf`, добавляет количество в `bulletPoint`.
- `index.ts` — создаёт `StubProvider`; на каждый файл определяет язык (`detectLanguage`)
  и модальную легенду преамбулы (`findModalityLegend`) и вызывает `buildLeaves`.

**Путь данных.** чанки-позиции → `buildLeaves` (для каждой: ключи уровней + `enrichLeaf`
через провайдер) → список `CandidateLeaf` с провенансом (`chunkIds`).

**Проверка (на 3 примерах).** Fahrrad — **6 листьев**; Salzburg — 1; Christmas — 0.
Провайдер в логах — `deterministic-stub`. Итог: `leaves:7`.

**Связь с дальнейшим.** Листья пойдут в **consolidate** (слияние дублей по chunk-id),
затем в **assemble** (сборка 3-уровневого дерева из ключей L1/L2) и **validate**.

## Слайс 8 — consolidate (слияние дублей, exact-id)

**Зачем.** Это способность, которую бриф ценит **выше всего**: одно требование, упомянутое
в разных местах, должно стать **одним** листом со всеми `chunkIds`. Здесь закладываем сам
механизм в самом точном варианте (exact-id); правила-скоуп и эмбеддинги+LLM-верификатор
добавим отдельными слайсами.

**Что сделали (файлы и функции).**
- `pipeline/consolidate.ts`, функция `consolidate(leaves)`: группирует листья по ключу

## Слайс 9 — assemble (сборка дерева) + запись output

**Зачем.** Собрать финальную структуру — 3-уровневое дерево `ProcurementMatchDeliverable` —

## Слайс 10 — validate (Zod-схема + аудит покрытия)

**Зачем.** Доказать, что вывод не только корректной формы, но и **полный**: ни одно реальное

**Итог.** Базовый конвейер работает от папки PDF до валидного дерева с отчётом. Дальше —

## Слайс 11 — реальный DeepSeek за интерфейсом + учёт токенов

**Что сделали (файлы и функции).**

**Путь данных.** тот же, но `provider.enrichLeaf` теперь ходит в API DeepSeek; ответ
валидируется Zod и превращается в поля листа; расход токенов копится и логируется в конце.

этот путь проверяется вручную.)

**Связь с дальнейшим.** Провайдер готов; следующий крупный шаг — **грамматика Salzburg**

## Слайс 12 — грамматика Salzburg (room-коды)

**Зачем.** Крупнейший прирост recall. Salzburg держит позиции в номерах вида `GU.20.05.01.01`
(этаж/комната.Обергруппа.подгруппа.порядковый), которые GAEB-грамматика не ловит — до этого
из ~1600 позиций распознавалась 1.

**Ключевая находка.** В строке позиции код **склеен с заголовком без пробела**:
`GU.20.04.01.03Labor- Unterschrank`. У групп пробел есть (`GU.20.05 Hängeschränke`). Поэтому
грамматика позиции не требует пробела после кода, а группы — требует; уровень различается по
числу сегментов (4 — позиция, 1–2 — группа).

**Что сделали (файлы).**
- `pipeline/parseLv.ts` — добавлены `RE_POSITION_ROOM` (`префикс + 4×.NN`, без обязательного
  пробела) и `RE_GROUP_ROOM` (`префикс + 1–2×.NN`, с пробелом). Префикс — 2 символа: буква +
  буква/цифра (`GU`, `G0`, `G1`), что заодно отсекает числовые коды преамбулы. Обе грамматики
  работают параллельно с GAEB; форматы не пересекаются (GAEB — цифры, room — с буквы).
  Существующее выведение уровней в `leaves` (`GU` → L1, `GU.20` → L2) подходит без изменений.

**Проверка (на 3 примерах).**
- Fahrrad — **6 позиций** (GAEB не задет).
- Salzburg — **1341 позиция** (было 1), 21 с «oder gleichwertig».
- Christmas — **0** (проза, ждёт свой слайс).
- **Консолидация впервые сработала по-настоящему:** `1347 → 1043, merges:304` — повторяющиеся
  по комнатам позиции слились в один лист с объединённым провенансом.
- Дерево: `level1:11, level2:105, level3:1043`; схема валидна, покрытие 100%, 0 сирот.

**Честное ограничение.** 1341 из ~1599 — восстановлено ~84%. Остаток (позиции с иным
форматированием, ~4 строки легенды-примера) — предмет доводки; recall уже вырос в ~1340 раз.
Подписи L1/L2 для Salzburg пока фолбэк-ные (`Title GU`, `Group GU.20`) — реальные названия
Обергрупп лежат в легенде преамбулы, их разбор — отдельная полировка.

**Связь с дальнейшим.** Дальше — прозаический путь Christmas (извлечение по заголовкам/спискам),
привязка аннексов и OCR; плюс доводка recall и подписей Salzburg.

## Слайс 16 — OCR битых страниц

**Зачем.** Quality-gate уже помечал битые страницы Christmas (сломанные лигатуры «tesng»).
Теперь их **чиним**: перечитываем со страницы-картинки, минуя сломанную шрифтовую таблицу.

**Что сделали (файлы и функции).**
- `pipeline/ocr.ts`, функция `ocrPage(pdfPath, page, lang)`: `pdftoppm` рендерит страницу в PNG
  (300 dpi), `tesseract` распознаёт текст. Если утилит нет — возвращает `null` (не падаем,
  оставляем помеченный текст). Важный нюанс: режим **`--psm 6`** (единый блок текста) — без него
  tesseract разрывает колонку номеров и колонку текста, и нумерованный список рассыпается.
- `pipeline/extract.ts` — после quality-gate для каждой `corrupted`-страницы вызывает `ocrPage`,
  заменяет строки этой страницы OCR-текстом, переоценивает качество (`method:"ocr"`), логирует.

**Путь данных.** extract помечает страницу `corrupted` → `ocrPage` рендерит+распознаёт →
строки страницы заменяются чистым OCR-текстом → дальше всё как обычно (classify, parseProse).

**Проверка (на 3 примерах).** Christmas: страницы 2–3 → `re-read via OCR`, качество `clean`,
`corrupted:0`. Текст восстановлен: «Electric **testing**», «Install and **activate** 23 sets of
pea lights…». Требований по-прежнему **15**, но уже с корректным текстом. Fahrrad/Salzburg — без
OCR (нет битых страниц). Схема валидна, покрытие 100%.

**Зависимость.** OCR требует системных утилит `poppler` (`pdftoppm`) и `tesseract`. На macOS:
`brew install poppler tesseract`. Без них пайплайн работает, но битые страницы остаются
помеченными (не чинятся) — деградация мягкая, не падение.

**Связь с дальнейшим.** Без ключа остаётся доводка подписей/recall Salzburg; затем с ключом —
реальный LLM-обогащение и эмбеддинг-консолидация с верификатором; финал — faithfulness и README.

## Слайс 17 — доводка Salzburg (recall + названия)

**Зачем.** Поднять полноту и качество дерева на большом тендере: добрать пропущенные позиции
и заменить фолбэк-подписи `Group GU.20` реальными названиями подгрупп.

**Найденная проблема (recall).** Chrome-детектор в classify **съедал повторяющиеся позиции**:
одинаковые шаблоны позиций повторяются по комнатам, их нормализованный вид совпадает, и детектор
принимал их за колонтитул. Из-за этого терялось ~170 позиций.

**Что сделали (файлы).**
- `pipeline/classify.ts` — строки, начинающиеся с кода позиции/группы (`CODE_LINE`), больше
  **никогда не считаются chrome**, даже если повторяются. Колонтитулы так не начинаются.
- `pipeline/parseLv.ts` — отсев 4 строк-легенды преамбулы (пример нумерации `GU.59… H laufende
  Nr…`), чтобы они не попадали в позиции.
- `pipeline/leaves.ts` — для room-кодов уровень L2 берётся как **подгруппа** (3-частный ключ

## Слайс 18 — устойчивость LLM к масштабу (параллельность + ретраи)

**Зачем.** На Fahrrad LLM-обогащение — 6 вызовов, а на Salzburg — ~1527 подряд: без
параллельности это очень долго, а без ретраев любой rate-limit (429) роняет прогон. Это
инфраструктура под масштаб — новых требований она не добавляет (это следующие слайсы).

**Провайдер-агностично.** Тот же интерфейс работает с DeepSeek и Groq: провайдер OpenAI-
совместим, поэтому смена сервиса — это только `env/.env` (ключ + `BASE_URL` + модель), без
правок кода. Проверено на Groq (`llama-3.3-70b-versatile`): 6 позиций Fahrrad обогащены,
учёт токенов работает.

**Что сделали (файлы).**
- `pipeline/leaves.ts` — `mapLimit`: обогащение идёт **пулом с ограничением параллельности**
  (сохраняя порядок). Плюс общий **бюджет LLM-вызовов**: сверх лимита — детерминированный stub.
- `llm/deepseek.ts` — `jsonCompletion` получил **ретраи на 429/5xx** с экспоненциальной паузой
  и учётом заголовка `Retry-After` (до 4 попыток).
- `config.ts` — флаги `--llm-concurrency N` (по умолчанию 6) и `--llm-max N` (сколько листьев
  максимум обогащать моделью за прогон; остальное — stub). Полезно, чтобы тестировать LLM на
  большом тендере без тысяч вызовов.

**Проверка.** Typecheck чист. Stub-путь (без `--use-llm`) не изменился: дерево **идентично**
прежнему (1094 листа, схема валидна) даже при `--llm-concurrency 8 --llm-max 3` — параллельность
не ломает детерминизм. Ретраи/параллельность на живом API проверяются на машине с сетью.

**Как гонять большой тендер с LLM безопасно.**
```
npm start -- --input .. --use-llm --llm-max 20 --llm-concurrency 1
```
(20 листьев через модель, остальное stub; 429 гасятся ретраями.)

**Важно про rate-limit.** У бесплатного тарифа Groq лимит — **токены в минуту (TPM 12000)**,
это очень мало. При таком лимите **высокая параллельность вредит**: залп из 8 запросов мгновенно
выжирает минутный бюджет и вызывает каскад 429. Поэтому на Groq free tier используйте
`--llm-concurrency 1` (последовательно, с само-троттлингом через ретраи, которые honor'ят
подсказку Groq «try again in Xs»). Массовое обогащение (весь Salzburg, ~1527 вызовов) на
бесплатном тарифе занимает ~час — для этого нужен платный тариф или рабочий ключ DeepSeek;
stub остаётся дефолтом для масштаба. Ретраи расширены до 8 попыток с backoff до 30с.

**Связь с дальнейшим.** Инфраструктура готова. Теперь — прирост recall: LLM-извлечение
обязательств из **преамбулы** и **прозы без номеров** (то, чего пока нет в дереве), затем
эмбеддинг-консолидация с верификатором; финал — faithfulness и README.

## Слайс 21 — семантическая консолидация с LLM-верификатором

**Зачем.** Exact-id консолидация ловит только одинаковые `bulletPoint`. Полная версия того,
что бриф ценит выше всего: находить **near-дубли** одного требования и подтверждать каждое
слияние LLM'ом.

**Что сделали (файлы).**
- `pipeline/consolidateSemantic.ts`, `consolidateSemantic(leaves, provider, budget)`:
  - кандидатов ищем **лексической близостью** (Jaccard по токенам) — провайдер-агностичная
    замена эмбеддингам, без зависимости от embedding-API;
  - **блокировка по Level-2 (подгруппе)**: сливаем только внутри одной подгруппы. Это осознанно
    **не трогает** одинаковые позиции, повторяющиеся по комнатам (разные подгруппы) — это разные
    количества, а не одно требование;
  - каждое предложенное слияние подтверждает `provider.verifyMerge` — **LLM как точный фильтр**;
  - бюджет вызовов (`--merge-max`, дефолт 60); на stub — **no-op** (`verifyMerge` всегда false).
- `index.ts` — стадия запускается только при `--use-llm`, после exact-id консолидации.

**Проверка (offline).** Без `--use-llm` стадия не запускается (нет лога `semanticMerge`), вывод
не меняется (1114 листьев, схема валидна, покрытие 100%). Реальные слияния — на LLM-прогоне.

**Честное ограничение.** Лексическая близость — это упрощённая замена эмбеддингам, а блокировка
по подгруппе намеренно консервативна: дальнюю кросс-документную привязку («стр. 60 + стр. 382 +
аннекс») мы **не** делаем автоматически — это риск ложных слияний, и в проде такое требует
настоящих эмбеддингов + ревью человеком. Здесь мы безопасно чистим near-дубли внутри подгруппы
и демонстрируем механизм verifier'а, который BOND ценит.

**Связь с дальнейшим.** Финал — faithfulness-проверка (сэмпл листьев: следует ли описание из
чанков) + настоящий README (cold start, design-решения, честные слабости) + закоммиченный
пример вывода для сдачи.
